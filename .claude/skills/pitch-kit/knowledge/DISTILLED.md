# Pitch-Kit Distilled — load this FIRST

One-page operating sheet distilled from judging-criteria, winning-patterns,
demo-psychology, pitch-strategy, and submission-guidelines. Open the full
files only when a specific section needs depth. The Day-0 rubric extraction
ALWAYS overrides the typical weights below.

## Scoring (typical weights — the real rubric overrides)
Technical/implementation 20–35% · Innovation 15–30% · Impact 15–25% ·
Design/UX 10–20% · Pitch 15–25% · Track/sponsor alignment often decides
track prizes. Judge types: technical → working code; business → market;
domain → accuracy; sponsor → their API. Calibrate emphasis to the panel.

## The ten rules that win
1. Demo path first: one 90-second sequence; hardcode, mock, skip auth —
   anything to make that sequence flawless. Polish expands outward from it.
2. Exactly ONE wow moment, landed before 40% of demo time; pause 2–3s after
   it (the silence does the selling).
3. Open with a felt problem in ≤10 seconds, second person, no statistics —
   first impressions form in 10s and barely move afterward.
4. Show the hard part actually working; a live URL beats any mockup.
5. Contrast: show the broken world before the fix — same result lands harder.
6. Map every feature to a rubric criterion; speak the sponsor's language.
7. Narrate outcomes, not actions ("Now she can…" not "I'm clicking…");
   narrate every loading state — unexplained pauses create judge anxiety.
8. Anchor the wow before it happens: "What you're about to see is…"
9. Close with a question, not "thank you" — it keeps judges thinking.
10. Peak-end rule: one extraordinary moment + a strong close beats
    consistently-good-but-flat.

## Pitch skeleton + timing (3-min; scale proportionally)
Hook+Problem 30s → Solution (one sentence, "so that <outcome>") 20s →
Demo 80s → Tech (one slide, name the sponsor API) 15s → Impact 15s →
Team ("why us") 10s → Close 10s. Demo never exceeds 50% of the pitch.
Practice to 80% of the time limit. Every spoken line ≤15 words.

## Q&A reflexes
"Why not [competitor]?" → acknowledge, then differentiate. "Does it scale?"
→ today's real limit + next step. "Is it real?" → point at the live demo,
honest real-vs-mocked split. Unknown → "We don't know yet — here's how we'd
find out." Prep ≥5 adversarial questions before presenting (/judge-sim).

## Submission (judges score what they can SEE)
Draft in the first 4h · all fields ≥2h before deadline · video ≥1h before ·
submit 30–60 min early — NEVER in the last 15 min (platforms choke).
Video 90s–3min, ≥720p, project name visible in the first 3s.
README: screenshot/GIF, stack, one-command setup, live link, video link,
team. Description order: inspiration → what it does → how we built it →
challenges → proud of → learned → what's next. Every claim verifiable from
the demo or the repo.

## Environment checklist (before any judged run)
Notifications off · test data loaded · zoom 125–150% · large cursor ·
name + tagline visible on the first screen · presenting late in the session?
open higher-energy and shorter, not longer · demo owner has final call.
