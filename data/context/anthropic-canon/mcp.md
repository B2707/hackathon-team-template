## Participants & the end-to-end request flow

- MCP has three named participants: **Host** (the AI app — Claude Desktop, Claude Code, VS Code, Cursor), **Client** (a component inside the host, one per server, holding a single dedicated connection), and **Server** (a program exposing tools/resources/prompts to a client — e.g. GitHub, filesystem, Sentry servers) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- Cardinality is 1 host : N clients : N servers — the host spins up one dedicated client per server, so connecting to 3 servers means the host holds 3 clients (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- One client per server (not one shared client) buys isolation: independent connection lifecycle, independently negotiated capabilities, independent state; a crash or slowness in one server can't affect others, and each server only ever sees its own client (a clean security boundary) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- "Server" is a role, not a location or size — it can be a tiny local script or a hosted cloud service; what makes it a server is that it serves tools/resources/prompts to a client. Local server = subprocess on the host machine (usually host-launched), stdio transport, single client; remote server = another machine over the network, Streamable HTTP transport, many concurrent clients (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- The standardized client-server contract lets any host use any server without either side knowing the other's internals — this is the basis of MCP's scalability (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- Beginner mistake to avoid: calling the client "your server," or merging host and client into one concept — keep the three roles distinct (https://www.anthropiccertifications.com/courses/introduction-to-mcp/architecture-host-client-server).
- Five players in every interaction: USER, APPLICATION (the host), MCP CLIENT, MCP SERVER, and CLAUDE (the LLM) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/end-to-end-request-flow).
- End-to-end flow (11 steps, condensed): user query → app asks the client for tools → client sends `tools/list` → app sends query+tools to Claude → Claude returns a `tool_use` naming a tool → app has the client run `tools/call` → server hits the real external API → the result flows back server→client→app→Claude → Claude composes the final natural-language answer (https://www.anthropiccertifications.com/courses/introduction-to-mcp/end-to-end-request-flow).
- Critical architectural point: Claude never talks to the MCP server directly — it only emits a `tool_use` request; the application (via the client) actually executes it and feeds the result back. This decouples the model from infrastructure and lets the app insert approval gates, logging, or policy checks at the execution step (https://www.anthropiccertifications.com/courses/introduction-to-mcp/end-to-end-request-flow).
- Role split: the app orchestrates the whole loop; the MCP client only translates "list/run tool" into JSON-RPC `tools/list`/`tools/call` and back; the MCP server implements the integration (definitions + execution against the real service); Claude only decides which tool to call and writes the final answer (https://www.anthropiccertifications.com/courses/introduction-to-mcp/end-to-end-request-flow).

## Two layers: data (JSON-RPC) and transport

- MCP has two layers: the **data layer** (inner — WHAT is exchanged: JSON-RPC 2.0 messages, lifecycle, capabilities, the primitives, notifications) and the **transport layer** (outer — HOW messages travel: connection setup, framing, auth). Separating them makes MCP transport-agnostic — you write the server once and pick the transport at run time (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc).
- The data layer is JSON-RPC 2.0 with three message shapes: **request** (`id` + `method` + optional `params`, expects a response), **response** (correlated back by the same `id`, carries `result`), and **notification** (`method`, no `id`, no response — fire-and-forget, e.g. `notifications/tools/list_changed`) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc).
- Using an existing standard (JSON-RPC) means SDKs in any language speak the same wire format; you rarely hand-write these messages, but recognizing them matters for debugging (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc).
- Servers can push notifications when state changes without being asked — e.g. `notifications/tools/list_changed` triggers the client to re-fetch the tool list, keeping the model's view of capabilities current without polling; notifications are capability-gated, so a server only sends `list_changed` if it declared that capability at init (e.g. `"tools": {"listChanged": true}`) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc).

## Transports: stdio and Streamable HTTP

- MCP defines exactly **two** official transports today: **stdio** and **Streamable HTTP**. The transport layer handles connection establishment, message framing (delimiting one JSON-RPC message from the next), and — for networked transports — authentication (https://www.anthropiccertifications.com/courses/introduction-to-mcp/transports-stdio-and-http).
- **SSE is NOT a separate transport** — it is an optional streaming mechanism *inside* Streamable HTTP; older "HTTP+SSE transport" docs mean today's Streamable HTTP. WebSockets are not a transport either. (Common exam trap.) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/transports-stdio-and-http and https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips)
- **stdio**: uses OS stdin/stdout between processes on the same machine; the host launches the server as a subprocess, writes JSON-RPC to stdin, reads replies from stdout. No network/ports/HTTP overhead — the fastest option and the default for local servers. Best for local integrations (filesystem, local DB, local scripts); typically serves a single client; credentials come from environment variables (the OAuth flow does NOT apply to stdio) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/transports-stdio-and-http).
- **Streamable HTTP**: HTTP POST for client→server messages, with optional SSE to stream partial results back over a single response (useful for long-running tools with incremental output); works across the network; supports standard HTTP auth (bearer tokens, API keys, custom headers), with OAuth 2.1 recommended for obtaining tokens; serves many clients concurrently (https://www.anthropiccertifications.com/courses/introduction-to-mcp/transports-stdio-and-http and https://www.anthropiccertifications.com/courses/introduction-to-mcp/remote-servers-and-streamable-http).
- Choosing a transport is driven by where the server runs, client count, credential handling, and network overhead: stdio for local single-client dev tools; Streamable HTTP for remote, multi-client SaaS servers (https://www.anthropiccertifications.com/courses/introduction-to-mcp/transports-stdio-and-http).
- Going remote is a one-line change on a FastMCP-style server: `mcp.run(transport="streamable-http")` instead of stdio — tool/resource/prompt code is untouched because of the two-layer design (https://www.anthropiccertifications.com/courses/introduction-to-mcp/remote-servers-and-streamable-http).
- Operational differences remote vs local: many concurrent clients (vs one); an independent long-running service (vs a host-launched subprocess); credentials via HTTP auth/OAuth tokens (vs env vars); optional SSE streaming; can run statelessly per request — a subset of MCP that suits serverless — (vs stateful per-process) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/remote-servers-and-streamable-http).
- Client-side, connecting to a remote server swaps `stdio_client` for `streamable_http_client("https://mcp.example.com/mcp")`; `ClientSession` and its API (`initialize`, `list_tools`, `call_tool`, `read_resource`, `get_prompt`) stay identical regardless of transport (https://www.anthropiccertifications.com/courses/introduction-to-mcp/remote-servers-and-streamable-http).
- Because a remote server is reachable by anyone on the network, OAuth-based authentication/authorization becomes essential (https://www.anthropiccertifications.com/courses/introduction-to-mcp/remote-servers-and-streamable-http).

## Connection lifecycle & capability negotiation

- Every MCP connection moves through three phases: **initialization** (negotiate capabilities), **operation** (list/call/read/notify), **shutdown** (clean teardown) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- The initialization handshake is stateful: (1) client sends `initialize` with its `protocolVersion` + `capabilities`, (2) server replies with its own `protocolVersion` + `capabilities`, (3) client sends `notifications/initialized` to signal readiness — only then does normal operation begin (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc and https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- The `capabilities` object is a contract: each side promises only what it declares, and the peer must never call a method for an undeclared capability — e.g. a server declaring `"resources": {}` (no sub-keys) supports resources with no special sub-features, while omitting `prompts` means the client must not call `prompts/list` (https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- `protocolVersion` is a date string (e.g. `2024-11-05`, `2025-03-26`, `2025-06-18`), not a semantic version — each date is a spec snapshot. Peers negotiate a mutually-supported version; if they can't agree, the connection must be terminated rather than risk mismatched behavior (https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- Shutdown differs by transport: for stdio, close the streams / end the subprocess; for HTTP, end the session (https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- **Golden rule**: never call a method for a capability the peer didn't advertise (e.g. calling `prompts/list` against a server that never declared `prompts` is a protocol error). SDKs help enforce this, but understanding it prevents a whole class of bugs (https://www.anthropiccertifications.com/courses/introduction-to-mcp/connection-lifecycle-and-capabilities).
- Capability negotiation lets old and new implementations interoperate on a common subset — each declares what it supports and neither attempts operations the other can't handle (https://www.anthropiccertifications.com/courses/introduction-to-mcp/the-data-layer-json-rpc).

## The three server primitives

- **The single highest-yield fact — the control model**: Tools = **model**-controlled, Resources = **application**-controlled, Prompts = **user**-controlled. One placing question decides where any feature belongs: "who should trigger it — the model, the app, or the user?" (https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips and https://www.anthropiccertifications.com/courses/introduction-to-mcp/choosing-the-right-primitive)

### Tools — model-controlled actions

- Tools are executable functions the model calls to take actions (search, send, query, edit); each has a name, a description, and a typed JSON-Schema input. Model-controlled: the LLM decides when to call them from the user request/context (https://www.anthropiccertifications.com/courses/introduction-to-mcp/tools-model-controlled).
- Two protocol operations: `tools/list` (discover — returns an array of tool defs with schemas) and `tools/call` (execute — returns a content-array result) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/tools-model-controlled).
- Python SDK (FastMCP): decorate a function with `@mcp.tool(name=..., description=...)`; the SDK reads type hints and `pydantic.Field(description=...)` to auto-generate the JSON Schema — no hand-written schemas. Plain Python exceptions (e.g. `raise ValueError` on not-found) serve as error handling; an edit/action tool like `edit_document(doc_id, old_str, new_str)` models best practice with precise per-parameter descriptions and an explicit exact-match contract (https://www.anthropiccertifications.com/courses/introduction-to-mcp/tools-model-controlled).
- The model selects and fills tool calls almost entirely from name + description + parameter descriptions — vague descriptions cause wrong tool choice and malformed arguments; treat descriptions as part of the prompt (reinforces the tool-interface guidance in tools-and-mcp.md) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/tools-model-controlled).
- Safety: because tools act and change real-world state (DB writes, emails, payments), MCP emphasizes human oversight — approval dialogs (especially destructive actions), per-tool permission settings, per-conversation enable toggles, and activity logs. JSON Schema validates argument *shape*, not *intent* — high-stakes ops (refunds, deletes, transfers) need programmatic gates or human approval beyond a good description (https://www.anthropiccertifications.com/courses/introduction-to-mcp/tools-model-controlled).

### Resources — application-controlled read-only data

- Resources are read-only data sources (file contents, DB schemas, API docs, calendar entries) addressed by URI, behaving like HTTP GET endpoints — "resources are nouns, tools are verbs" (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Application-controlled: the host app's code decides when to read a resource and how to use it (populate UI, autocomplete, inject into a prompt) — the model never calls a tool to fetch one. Example: Claude's "Add from Google Drive" button (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Three methods: `resources/list` (direct resources), `resources/templates/list` (discover templated definitions), `resources/read` (retrieve contents + MIME type) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Direct resource = a fixed, unparameterized URI (e.g. `docs://documents`) declared with a `mime_type`; `@mcp.resource("docs://documents", mime_type="application/json")` returns any Python object and the SDK serializes it. Templated resource = a URI with a placeholder (`docs://documents/{doc_id}`); the SDK parses the parameter out of a matching request and passes it as a keyword argument, so one definition serves a whole family — raise a clear error (`ValueError`) if the id isn't found (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled and https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-server).
- `mime_type` tells the client how to parse returned bytes: `application/json` → structured data parsed into objects; `text/plain` → text as-is; `application/pdf`/other → binary file content (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Templated resources (and prompt arguments) support parameter completion — as a user types a `{param}` the client can suggest valid values, enabling discoverable "@mention"-style UX without knowing exact IDs (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Design rule: pick a resource when the request/response follows a URI-keyed retrieval pattern AND the app (not the model) knows in advance it needs the data (https://www.anthropiccertifications.com/courses/introduction-to-mcp/resources-app-controlled).
- Client-side reading powers "@mention" UX: when the user types `@report.pdf`, the app itself reads the resource and injects its contents into the prompt — no tool call/round trip, so the model already has the data. Two `resources/read` calls implement mentions: read the direct list resource to populate autocomplete, then read the templated resource once the user picks one (https://www.anthropiccertifications.com/courses/introduction-to-mcp/reading-resources-in-the-client).
- Client read pattern: `session.read_resource(AnyUrl(uri))` → take `result.contents[0]` → branch on `resource.mimeType` (`application/json` → `json.loads(resource.text)`; otherwise return `resource.text` as a raw string) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/reading-resources-in-the-client).
- Resources vs tools for fetching data: a tool *could* fetch a document, but that costs an extra round trip (Claude decides → app runs tool → result returns); reading as a resource front-loads the data into context, is faster, and keeps the app in control of what's included — choose a resource whenever the app already knows it needs the data (https://www.anthropiccertifications.com/courses/introduction-to-mcp/reading-resources-in-the-client).

### Prompts — user-controlled workflows

- Prompts are reusable, parameterized templates that package expert instructions for a common task — the author writes and tests the prompt once, and users invoke it on demand instead of reinventing wording. User-controlled: triggered by an explicit user action (slash command, button, menu pick), never automatically by the model or silently by the app (Claude's workflow buttons under the chat box are prompts) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-user-controlled).
- Two methods: `prompts/list` (discover prompts + their arguments) and `prompts/get` (retrieve the prompt with arguments interpolated into the actual message(s) sent to the model). Unlike `list_tools`/`list_resources`, `get_prompt` performs variable interpolation and returns ready-to-send messages, not a bare definition (https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-user-controlled and https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-in-the-client).
- Value vs free-form typing: consistency (reliable output regardless of phrasing), expertise (edge cases/best practices baked in), reusability (shared across client apps), and centralized maintenance (improve once, all clients benefit) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-user-controlled).
- SDK pattern: decorate with `@mcp.prompt(name=..., description=...)`, accept parameters (e.g. via `pydantic.Field`), and return `list[base.Message]` (e.g. `[base.UserMessage(prompt_text)]`) — not a plain string. Arguments become keyword args on the server, are interpolated into the template, and the resulting messages are returned to send to Claude; prompt arguments support completion (https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-user-controlled and https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-in-the-client).
- Prompts often orchestrate the other two primitives — a single tested, user-triggered workflow that reads resources and calls tools in a repeatable way (https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-user-controlled).

### Choosing & composing the primitives

- Decision table: Tools (model) give Claude new abilities it invokes autonomously (run code / call an API); Resources (app) pull data into the app for UI/context ("Add from Google Drive"); Prompts (user) offer predefined workflows triggered on demand (workflow buttons under the chat input) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/choosing-the-right-primitive).
- Real servers combine all three — e.g. a "plan-vacation" PROMPT the user triggers, calendar/preferences loaded as RESOURCES for context, then TOOLS to search flights/check weather/book a hotel. The course's document server: tools `read_doc`/`edit_document` (act), resources `docs://documents` + `docs://documents/{id}` (feed autocomplete/context), prompt `/format` (a tested reformatting workflow) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/choosing-the-right-primitive and https://www.anthropiccertifications.com/courses/introduction-to-mcp/prompts-in-the-client).
- Anti-patterns: a tool whose only job is to "fetch some data for the UI" should be a resource; a long instruction users keep retyping should be a prompt (https://www.anthropiccertifications.com/courses/introduction-to-mcp/choosing-the-right-primitive).
- Method mapping quick reference: Tool → `tools/list`, `tools/call`; Resource → `resources/list`, `resources/templates/list`, `resources/read`; Prompt → `prompts/list`, `prompts/get` (https://www.anthropiccertifications.com/courses/introduction-to-mcp/choosing-the-right-primitive and https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).

## The three client primitives (server-requested)

- Client primitives flip the usual direction: instead of the server offering primitives to the client, the server requests something and the client fulfills it. Three exist: **Sampling** (`sampling/createMessage` — server asks the client's LLM for a completion), **Elicitation** (`elicitation/create` — server asks the end user for input/confirmation), and **Logging** (log/debug notifications sent to the client) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/client-primitives-sampling-elicitation-logging).
- The capability-negotiation rule applies in reverse: a server may only call a client primitive the client advertised during initialization (e.g. only call `elicitation/create` if the client declared `"elicitation": {}`) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/client-primitives-sampling-elicitation-logging).
- **Sampling** lets a server request a completion from the client's AI application without bundling its own model SDK or API key — the server stays model-independent (e.g. a server summarizing DB rows sends them via sampling and lets the host's Claude summarize). Because it spends the host's model budget and runs generations the user didn't directly type, well-behaved clients keep a human in the loop (show/approve sampling requests) — the protocol enables the capability, the client governs its use (https://www.anthropiccertifications.com/courses/introduction-to-mcp/client-primitives-sampling-elicitation-logging).
- **Elicitation** lets a server pause mid-operation to request a missing value or confirmation ("Which environment should I deploy to?", "Confirm: delete 3 records?"); the client renders the request to the user and returns the answer to the server (https://www.anthropiccertifications.com/courses/introduction-to-mcp/client-primitives-sampling-elicitation-logging).
- **Logging** lets a server send log messages to the client for debugging/monitoring — useful when the server's own console isn't visible (e.g. a remote deployment). Beyond the three client primitives, MCP has cross-cutting utilities: progress notifications for long-running operations and an experimental Tasks primitive for durable/deferred execution (https://www.anthropiccertifications.com/courses/introduction-to-mcp/client-primitives-sampling-elicitation-logging).

## Authorization & security

- MCP's authorization spec applies only at the transport level for HTTP-based transports and is optional overall (but should be followed if you support auth over HTTP). stdio servers do NOT use this OAuth flow — being local subprocesses, they read credentials straight from environment variables (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security).
- Authorization is based on OAuth 2.1 (plus related metadata / dynamic-client-registration standards). Role mapping: the MCP server = OAuth 2.1 resource server (validates tokens, does not issue them); the MCP client = the OAuth client acting on the user's behalf; a separate authorization server handles login and issues access tokens (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security).
- Discovery/auth flow: client calls the server with no token → server responds 401 + metadata → client runs the OAuth 2.1 + PKCE flow against the authorization server to get an access token → client retries the request with a `Authorization: Bearer` token (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security).
- **Five non-negotiable rules**: (1) **validate token audience** — the server MUST verify each token was issued specifically for it; (2) **no token passthrough** — never forward a client's token to a downstream API, use the server's own separate upstream token (passthrough enables the "confused deputy" attack); (3) clients MUST use **PKCE** (prevents auth-code interception/injection); (4) bearer tokens go only in the `Authorization: Bearer` header on every request, never in the URL query string; (5) **HTTPS everywhere** for auth endpoints, plus redirect URIs validated against pre-registered exact values (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security and https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).
- Token passthrough is the single most-emphasized rule — violating it causes the confused-deputy vulnerability, where a downstream service wrongly trusts a token it should not accept (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security).
- Defense-in-depth beyond tokens (tools take real actions and servers may be third-party code): treat all tool inputs/outputs as untrusted and validate/sanitize them; apply least privilege (grant a server only the access it truly needs); keep human approval on high-stakes model-controlled calls (refunds, deletes, transfers); use short-lived access tokens and rotate refresh tokens to limit leak blast radius; prefer official/audited servers, since a malicious server is code running with whatever access you grant it (https://www.anthropiccertifications.com/courses/introduction-to-mcp/authorization-and-security).

## Building & testing servers/clients (Python SDK + Inspector)

- Install via `uv add "mcp[cli]"`. Three steps to a complete server: create a `FastMCP("Name")` instance, register primitives via decorators, call `mcp.run()`. One server can host all three primitive types together (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-server).
- SDK benefits: no manual JSON Schema (type hints + `Field` descriptions generate it), automatic argument validation from type hints, error handling via normal Python exceptions, automatic registration via decorators. Official SDKs exist for multiple languages (Python and TypeScript named explicitly) — same concepts everywhere: create server, register primitives, choose transport (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-server).
- Transport is a one-line choice at run time: `mcp.run(transport="stdio")` for local dev / Claude Desktop vs `mcp.run(transport="streamable-http")` to serve remotely — the primitives themselves don't change (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-server).
- In real projects you typically build either a client or a server, not both. A client has two layers: the SDK-provided `ClientSession` (the actual connection) and a thin wrapper class you write for usability and guaranteed cleanup (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-client).
- Connecting to a local stdio server: `StdioServerParameters(command="uv", args=["run", "mcp_server.py"])`, then `async with stdio_client(params) as (read, write): async with ClientSession(read, write) as session: await session.initialize()`. `initialize()` performs the capability-negotiation handshake; the async context managers ensure the subprocess and connection tear down cleanly (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-client).
- Two essential client methods wrap the session: `list_tools()` returns `result.tools` (definitions to send Claude) and `call_tool(tool_name, tool_input)` returns a `CallToolResult`. In the full loop the client never calls Claude directly — the app talks to Claude and uses the client purely to reach the server; sanity-check the client standalone (print the tool list) before wiring in Claude (https://www.anthropiccertifications.com/courses/introduction-to-mcp/build-an-mcp-client).
- The **MCP Inspector** is a browser-based tool that connects directly to your server so you can list and run every tool/resource/prompt by hand, with no client app — the fastest way to test and debug before wiring into a full application. Launch with `uv run mcp dev mcp_server.py`; it prints a local URL (typically `http://127.0.0.1:6274`) to open (https://www.anthropiccertifications.com/courses/introduction-to-mcp/testing-with-the-mcp-inspector).
- Inspector workflow: click Connect to run the `initialize` handshake (Disconnected → Connected), then use the Tools/Resources/Prompts tabs. Tools: List → select → fill fields → Run → verify status + returned data. Resources: separate direct vs template sections (templates require parameters); inspect the exact response body and MIME type a client would receive. Prompts: shows precisely which messages will be sent to Claude with arguments interpolated, so you verify wording before users rely on it (https://www.anthropiccertifications.com/courses/introduction-to-mcp/testing-with-the-mcp-inspector).
- Server state persists across Inspector calls, enabling multi-step workflow tests (run `edit_document`, then `read_doc_contents` to confirm the edit applied) and deliberate edge-case/error testing (e.g. an unknown doc id). The tight loop — change a tool, reconnect, run, see the result — catches schema mistakes, error-handling gaps, and bad descriptions before they surface inside a larger app (https://www.anthropiccertifications.com/courses/introduction-to-mcp/testing-with-the-mcp-inspector).

## Going to production: reliability & debugging

- Passing the Inspector locally is not the same as production-ready — real usage adds unexpected inputs, flaky upstream services, concurrent clients, and problems you can't directly observe. Production usually raises two bars at once: serving many concurrent clients (remote, Streamable HTTP) and running unattended with no one watching the console — both demand stronger error handling and observability (https://www.anthropiccertifications.com/courses/introduction-to-mcp/production-and-debugging).
- Robust-tool checklist (tools get model-controlled, unpredictable arguments): validate inputs beyond the JSON schema and raise clear, specific errors the model can recover from; explicitly distinguish "no results found" (a valid, non-retryable answer) from "the call failed" (may warrant a retry) — conflating them makes the model retry forever or give up incorrectly; keep outputs concise (trim verbose upstream responses to save context tokens); make operations idempotent so a retried call can't double-charge or double-send; set timeouts on upstream calls so one slow dependency can't hang the whole request (https://www.anthropiccertifications.com/courses/introduction-to-mcp/production-and-debugging).
- Destructive/costly model-controlled tools (deletes, refunds, transfers, bulk emails) need a human-approval step or a hard programmatic gate — a well-written tool description is explicitly NOT a safety control (https://www.anthropiccertifications.com/courses/introduction-to-mcp/production-and-debugging).
- Debugging a server you can't watch directly: reproduce the issue in the Inspector in isolation → observe via MCP's logging primitive (surfaces server-side events to the client) plus your own server-side structured logs/metrics → harden the code (validate, gate, add timeouts) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/production-and-debugging).
- **Deployment checklist** before exposing a server to real users: (1) **Transport** — stdio for local/single-user, Streamable HTTP for remote/multi-user (stateless mode for serverless); (2) **Auth** — OAuth 2.1 with token-audience validation and NO token passthrough, secrets via env vars, never hard-coded; (3) **Reliability** — input validation, clear errors, upstream timeouts, idempotency, concise outputs; (4) **Safety** — human approval / programmatic gates on destructive or costly tools, least privilege; (5) **Observability** — server logs/metrics plus MCP logging to the client, reproduce issues in the Inspector; (6) **Versioning** — advertise your protocol version and test against the specific clients (hosts) you intend to support (https://www.anthropiccertifications.com/courses/introduction-to-mcp/production-and-debugging).

## Consuming MCP in Claude Desktop & Claude Code

- Most users meet MCP as consumers of an existing host (Claude Desktop, Claude Code, VS Code, Cursor) that already embeds the client machinery — adding a server is configuration, not coding; the host instantiates a client automatically, runs the `initialize` handshake, and discovers tools/resources/prompts (https://www.anthropiccertifications.com/courses/introduction-to-mcp/using-mcp-in-claude).
- Claude Desktop registers local stdio servers under `mcpServers`, each with `command` and `args` — e.g. `"documents": {"command": "uv", "args": ["run", "mcp_server.py"]}`, or the official filesystem server via `npx -y @modelcontextprotocol/server-filesystem /path`. On next launch it starts each as a subprocess and creates a client; sensitive tool use is often gated behind a first-use approval prompt (https://www.anthropiccertifications.com/courses/introduction-to-mcp/using-mcp-in-claude).
- Claude Code is also an MCP host with two config scopes: **Project** scope (checked into the repo, shared with the team — Jira/GitHub/internal APIs) and **User** scope (personal, not shared — experimental/personal servers). Server prompts can surface as Claude Code slash commands. (Harness-side `.mcp.json` scoping specifics are in tools-and-mcp.md.) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/using-mcp-in-claude)
- Credentials rule: never hard-code tokens in a shared/project config — use environment-variable expansion, since a committed token leaks to everyone with repo access (https://www.anthropiccertifications.com/courses/introduction-to-mcp/using-mcp-in-claude).
- Primitive-to-UI mapping: Tools → Claude autonomously calling server functions mid-conversation (approval-gated for sensitive ones); Resources → app features like an "attach from…" context picker; Prompts → user-triggered workflows surfaced as slash commands or buttons (https://www.anthropiccertifications.com/courses/introduction-to-mcp/using-mcp-in-claude).

## Mental model & common traps (quick reference)

- MCP mental model in five numbers: **3** participants (Host, Client [1 per server], Server); **2** layers (data = JSON-RPC 2.0/lifecycle/primitives, transport); **2** transports (stdio local, Streamable HTTP remote); **3** server primitives (tools/resources/prompts); **3** client primitives (sampling/elicitation/logging) (https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).
- Core methods to memorize: `tools/list` + `tools/call`; `resources/list` (+ `resources/templates/list`) + `resources/read`; `prompts/list` + `prompts/get`; `initialize` + `notifications/initialized` (https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).
- Common traps: treating MCP as a replacement for tool-use (it's complementary — MCP supplies the tools, tool-use calls them; see tools-and-mcp.md); conflating host and client (Host → Client [1 per server] → Server); listing SSE/WebSockets as transports; putting app-already-known data fetches into a tool instead of a resource; skipping capability-negotiation checks before calling a method; passing a client's auth token through to a downstream API (https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).
- Deepening steps: build a real server for a daily-use tool, run it in Claude Desktop, convert it from stdio to Streamable HTTP and add OAuth, review the official reference servers (filesystem, GitHub), and read the 2025-06-18 spec for exact wire-level schemas (https://www.anthropiccertifications.com/courses/introduction-to-mcp/course-review-and-exam-tips).
