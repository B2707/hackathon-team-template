# Agent Architecture & Orchestration

Domain 1 (27%). The agentic loop, tool-use mechanics, multi-agent
coordinator-subagent design, task decomposition, and session state.

## Agentic loop lifecycle & control

- The loop: send request → inspect `stop_reason` → if `tool_use`, execute the tool(s), append the assistant message AND tool results to history → re-invoke → repeat until `end_turn` (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- `stop_reason` is the only reliable loop-control signal: continue while `tool_use`, terminate on `end_turn` (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- Anti-patterns: parsing natural-language "I'm done" cues to decide termination; treating presence of assistant text as completion; using an iteration cap as the primary stop mechanism (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- A max-iteration cap is a safety guardrail against runaway loops — a backup, not the control mechanism; iteration count should scale with task complexity (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- `stop_reason` values: `end_turn` (present to user), `max_tokens` (output may be truncated — handle/raise limit), `stop_sequence` (parse up to it), `tool_use` (run tools, return results) (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- Log each iteration for debuggability (https://www.anthropiccertifications.com/learn/agentic-architecture/agentic-loop-lifecycle).
- Division of labor: the model only *proposes* the next action (which tool, which inputs — model-driven decision-making); your loop code *executes* it and decides whether to continue or stop. Because the model never runs anything itself, hard rules placed in the loop code cannot be talked around (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/agentic-loops).
- The Messages API is stateless — the conversation history you resend IS the agent's only memory. Every turn, append the assistant's tool-use message AND the tool result, then resend the whole history; skip the append and the model can't see what the tool returned, so it re-requests the same tool indefinitely (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/agentic-loops).
- Additional `stop_reason` values a production loop must recognize: `pause_turn` (a built-in server tool hit its internal step limit — send the response straight back so the model continues) and `refusal` (safety decline — stop; newer models expose a `stop_details.category`). Newer Sonnet models can also return `model_context_window_exceeded` (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/agentic-loops).
- Fourth termination anti-pattern: forcing `tool_choice: "any"` so the model must always call a tool — it can then never emit `end_turn`, producing a loop that cannot stop (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/agentic-loops).
- Gotcha: adding an extra text note right after a tool result teaches the model that pattern and it may end turns early with a near-empty reply. Send the tool result on its own; if a reply still comes back empty, send a fresh "please continue" user message instead of resending the empty one (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/agentic-loops).

## Tool-use flow & mechanics

- Structured cycle: define tools in the request → Claude returns a `tool_use` block (`id`, `name`, `input`) with `stop_reason: tool_use` → your app executes → you return a `tool_result` block referencing the `tool_use` id → Claude answers or calls more tools. Mnemonic: define → select → execute → return (https://www.anthropiccertifications.com/learn/agentic-architecture/tool-use-flow).
- Parallel tool use: Claude can request several tools in one turn; run them and return all results together before the next call to cut round-trips (https://www.anthropiccertifications.com/learn/agentic-architecture/tool-use-flow).
- Prefer model-driven tool selection (Claude reasons the next tool from context) over hard-coded decision trees — it adapts to novel situations; use pre-configured sequences only where determinism outweighs flexibility (https://www.anthropiccertifications.com/learn/agentic-architecture/tool-use-flow).

## Coordinator-subagent (orchestrator / hub-and-spoke) pattern

- One coordinator owns all routing, error handling, and result aggregation; subagents never talk to each other — everything routes through the hub, giving centralized observability and single-point error handling (https://www.anthropiccertifications.com/learn/agentic-architecture/coordinator-subagent).
- Subagents run with isolated context — they do NOT inherit the coordinator's conversation history; the coordinator decides exactly what each subagent sees (https://www.anthropiccertifications.com/learn/agentic-architecture/coordinator-subagent).
- Dynamic routing: analyze the query and invoke only the subagents needed — a simple query may hit one; do not force everything through the full pipeline (https://www.anthropiccertifications.com/learn/agentic-architecture/coordinator-subagent).
- Iterative refinement: the coordinator inspects synthesis output for gaps, re-delegates targeted queries, and re-runs synthesis until coverage is adequate (https://www.anthropiccertifications.com/learn/agentic-architecture/coordinator-subagent).
- Benefits: centralized visibility, consistent recovery, information control (filter what each agent receives), and swappable downstream agents (https://www.anthropiccertifications.com/learn/agentic-architecture/coordinator-subagent).

## When multi-agent pays off — and when a single agent wins

- Hard economics of the orchestrator-worker pattern: multi-agent (Opus-4 lead + Sonnet-4 subagents) beat single-agent Opus-4 by **+90.2%** on Anthropic's internal research eval, strongest on breadth-first queries — but a multi-agent run burns **~15× the tokens** of a chat interaction (a standard single agent ~4×), and token usage ALONE explains **~80% of the performance variance**. So most of the gain is just "doing more work": it is a deliberate cost-for-capability trade, not a free win (https://www.anthropiccertifications.com/courses/introduction-to-subagents/multi-agent-orchestration).
- Choose by the SHAPE of the work, not its difficulty. Multi-agent when it is heavily parallelizable, the information exceeds one context window, it is breadth-first (many independent directions), or it needs many complex tools; single agent when each step depends on the previous, everything fits one context, the task is narrow-and-deep, or the agents would share heavy state — this includes MOST coding, whose interdependent subtasks fail the "does the intermediate work matter?" test. Agents issuing 3+ tool calls in parallel can cut research time up to ~90% (https://www.anthropiccertifications.com/courses/introduction-to-subagents/multi-agent-orchestration).
- Scaling heuristic: a simple query = 1 agent making 3-10 tool calls; complex research = 10+ subagents. Give each subagent brief four elements — objective, output format, tool/source guidance, explicit boundaries — start broad then narrow, and keep a human in the loop to catch source-selection bias and hallucinations on unusual queries (https://www.anthropiccertifications.com/courses/introduction-to-subagents/multi-agent-orchestration).

## Subagent invocation, Task tool, AgentDefinition

- Subagents are spawned via the `Task` tool; each `Task` call launches one independent run. The coordinator's `allowedTools` MUST include `"Task"` or it cannot delegate at all (https://www.anthropiccertifications.com/learn/agentic-architecture/subagent-invocation).
- Spawn subagents in parallel by emitting multiple `Task` calls in a single coordinator response — not spread across turns (https://www.anthropiccertifications.com/learn/agentic-architecture/subagent-invocation).
- Context isolation is absolute: assume a subagent knows nothing beyond its prompt. Pass complete prior findings explicitly; use structured formats that keep content separate from metadata (URLs, doc names, page numbers) so attribution survives (https://www.anthropiccertifications.com/learn/agentic-architecture/subagent-invocation).
- `AgentDefinition` configures each subagent type via three fields: **description** (coordinator uses it to decide when to invoke), **system prompt** (behavior/constraints), and **tool restrictions** (least privilege) (https://www.anthropiccertifications.com/learn/agentic-architecture/agent-definition-config).
- Write coordinator/subagent prompts as goals + quality criteria, not rigid step-by-step procedures, so the subagent adapts to the specific query (https://www.anthropiccertifications.com/learn/agentic-architecture/agent-definition-config).
- Don't summarize away details a downstream subagent needs — forward both web-search and document-analysis outputs to a synthesis agent intact (https://www.anthropiccertifications.com/learn/agentic-architecture/agent-definition-config).
- Naming: the spawning tool was renamed `Task` → `Agent` in Claude Code v2.1.63. The old name still works and the exam guide still says "Task"; some surfaces still emit `Task` (the init tools list, permission denials) while `tool_use` blocks emit `Agent`, so when detecting delegation match BOTH (https://www.anthropiccertifications.com/courses/introduction-to-subagents/subagents-in-the-agent-sdk).
- No nesting: the hierarchy is exactly one level deep — subagents have no Agent/Task tool and cannot spawn subagents, so never put Agent/Task in a subagent's tool list (this is why the read-only Plan subagent exists as the dedicated researcher) (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/subagent-invocation-context-passing).
- Exact context boundary: a subagent RECEIVES only its own system-prompt body + the Agent task prompt + project CLAUDE.md + its tool definitions; it does NOT receive the parent's conversation, the parent's tool results, the parent's system prompt, other subagents' results, its own prior invocations, or your preloaded skills (unless named in `skills:`). Only its final message returns, verbatim (https://www.anthropiccertifications.com/courses/introduction-to-subagents/subagents-in-the-agent-sdk).
- Scale-out cost: parallelism saves TIME, not TOKENS — every parallel subagent still returns its results into the main context, so many detailed returns can themselves flood it. Turn-by-turn Agent calls suit a few tasks per turn; for dozens-to-hundreds of agents (big migrations/audits) use the SDK's Workflow tool, which runs orchestration in a script OUTSIDE the conversation context (https://www.anthropiccertifications.com/courses/introduction-to-subagents/parallel-sequential-and-workflows).

## Claude Code subagents: config file, design levers, and anti-patterns

- A Claude Code subagent is a Markdown file at `.claude/agents/<name>.md`: YAML frontmatter (only `name` + `description` required; `tools`, `model`, `color` optional) and a body that IS the subagent's entire system prompt — it does not receive the main session's default system prompt (https://www.anthropiccertifications.com/courses/introduction-to-subagents/the-subagent-config-file).
- Omitting `tools` makes the subagent inherit ALL tools — least privilege requires setting it explicitly. Advanced frontmatter: `disallowedTools`, `permissionMode` (default/acceptEdits/plan/dontAsk/bypassPermissions), `skills` (preload skill content), `mcpServers`, `memory` (persistent cross-session directory), `maxTurns`, `effort`, `background`, `initialPrompt`, `isolation: worktree` (run in a temporary git worktree, auto-cleaned if it makes no changes), and per-subagent `hooks` (https://www.anthropiccertifications.com/courses/introduction-to-subagents/the-subagent-config-file).
- Model-resolution precedence (highest first): the `CLAUDE_CODE_SUBAGENT_MODEL` env var > a per-invocation model param > the subagent frontmatter `model` > the main-conversation model; `model` accepts an alias (haiku/sonnet/opus), a full ID, `inherit`, or omitted (= inherit) — pick the cheapest that clears the quality bar (https://www.anthropiccertifications.com/courses/introduction-to-subagents/choosing-model-tools-and-color).
- Tool compose order: `tools` is an allowlist and `disallowedTools` a denylist; if BOTH are set the denylist applies FIRST, then the allowlist resolves against the remainder (a tool named in both is removed). `memory` scopes are user (`~/.claude/agent-memory/`, all projects), project (`.claude/agent-memory/`, git-shareable — the recommended default), or local (https://www.anthropiccertifications.com/courses/introduction-to-subagents/choosing-model-tools-and-color).
- Gotcha: some tools NEVER reach a subagent even if listed, because they depend on the main UI/session — Agent (no nesting), AskUserQuestion, EnterPlanMode/ExitPlanMode (unless `permissionMode` is plan), ScheduleWakeup, WaitForMcpServers. Don't build a subagent that needs them (https://www.anthropiccertifications.com/courses/introduction-to-subagents/choosing-model-tools-and-color).
- The `description` does two jobs: it decides WHEN the orchestrator delegates AND it shapes the task prompt the orchestrator writes. Bake hand-off instructions into it ("you must tell the agent precisely which files to review") — vague descriptions guarantee vague delegations; "use proactively"/"use immediately after…" plus concrete example scenarios drive automatic delegation (https://www.anthropiccertifications.com/courses/introduction-to-subagents/descriptions-that-shape-input-prompts).
- Defining an output format in the subagent's system prompt is the single highest-impact reliability improvement: sections act as stopping points (prevents over-long open-ended runs), the main thread gets predictable structure, and a final explicit status/decision section forces a committed verdict (https://www.anthropiccertifications.com/courses/introduction-to-subagents/output-formats-and-obstacle-reporting).
- Add an "Obstacles Encountered" section (setup issues, workarounds, flags, dependency problems) to the required output — the subagent's conversation is discarded, so hard-won fixes survive the isolation boundary only if they're in the returned summary (https://www.anthropiccertifications.com/courses/introduction-to-subagents/output-formats-and-obstacle-reporting).
- Decision rule: does the intermediate work matter to the main thread? No → delegate (research/exploration, fresh-eyes code review, tasks needing a different system prompt); yes → keep it in the main thread (https://www.anthropiccertifications.com/courses/introduction-to-subagents/the-decision-rule-and-execution-modes).
- Fresh-context review is a delegation-only benefit: a thread that authored code reviews it poorly; a reviewer subagent sees the diff without authorship history and can carry team review standards in its prompt (https://www.anthropiccertifications.com/courses/introduction-to-subagents/when-subagents-shine).
- Subagent anti-patterns: (1) expert personas — "you are a Python expert" adds delegation overhead but no capability the main thread lacks; (2) sequential dependent pipelines (reproduce → debug → fix) — only a summary crosses each hand-off, so dependent context leaks away; (3) test-runner subagents — they hide the full failing output needed to debug, and performed worst in Anthropic's own testing. Delegation must buy isolation, parallelism, or a genuinely different prompt (https://www.anthropiccertifications.com/courses/introduction-to-subagents/subagent-anti-patterns).
- Execution modes: foreground blocks and passes permission prompts through interactively; background runs concurrently but auto-denies any tool call that would prompt (the call fails, the subagent continues) — a background permission failure is retried by re-running in the foreground. A fork inherits the full conversation, system prompt, tools, and model (deliberately dropping input isolation, reusing the parent prompt cache) — use when re-explaining costs too much or to try parallel approaches from one baseline (https://www.anthropiccertifications.com/courses/introduction-to-subagents/the-decision-rule-and-execution-modes).

## Built-in subagents, scope & precedence

- Three built-ins with fixed profiles: **Explore** (Haiku, read-only — Write/Edit denied — for file discovery/search, with quick / medium / very-thorough levels Claude can request), **Plan** (inherits the main model, read-only, research during plan mode), and **general-purpose** (inherits the main model, ALL tools, multi-step explore-and-act); helper agents statusline-setup and claude-code-guide also exist (https://www.anthropiccertifications.com/courses/introduction-to-subagents/built-in-subagents).
- Speed gotcha: Explore and Plan deliberately SKIP CLAUDE.md and the session's git status to stay fast/cheap — every other built-in and custom subagent loads both — and neither built-in can access skills at all. If a project rule (or a skill) must reach Explore/Plan, restate it in the delegation prompt (https://www.anthropiccertifications.com/courses/introduction-to-subagents/built-in-subagents).
- Subagent scope precedence (highest wins on a name collision): managed settings (org) > `--agents` CLI flag (session-only JSON, not saved to disk) > `.claude/agents/` (project) > `~/.claude/agents/` (user) > plugin `agents/`. Identity comes ONLY from the `name` frontmatter — both dirs are scanned recursively so subfolders are allowed but don't change identity, and two files with the same name in one scope silently drop one. Project subagents are found by walking UP from the working directory; `--add-dir` grants file access but is NOT scanned for subagents. Plugin subagents are namespaced (e.g. `my-plugin:review:security`) and, for security, cannot use `hooks`, `mcpServers`, or `permissionMode` (https://www.anthropiccertifications.com/courses/introduction-to-subagents/scope-and-precedence).

## Task decomposition: fixed vs adaptive, and narrow-decomposition risk

- Fixed sequential pipeline (prompt chaining): for predictable multi-aspect workflows with known stages and stable inputs/outputs (e.g., extract → validate → transform → load); quality comes from thoroughness (https://www.anthropiccertifications.com/learn/agentic-architecture/task-decomposition-strategies).
- Adaptive decomposition: for open-ended work where scope is unknown until you investigate and early findings reshape next steps (e.g., "add tests to a legacy codebase" — map structure, find high-impact gaps, then adapt); exploration precedes planning (https://www.anthropiccertifications.com/learn/agentic-architecture/task-decomposition-strategies).
- Attention dilution: a large single-pass input (e.g., a 50-file PR in one shot) makes the model miss issues buried in the middle. Fix with two-pass decomposition — per-file local analysis for focused attention, then a cross-file integration pass for interactions (https://www.anthropiccertifications.com/learn/agentic-architecture/task-decomposition-strategies).
- Diagnostic meta-rule: separate STRUCTURAL problems (fixed by how the work is split — per-item passes plus a cross-item integration pass) from CAPABILITY problems (fixed by a better model or prompt). Attention dilution and wobbling multi-item output are structural: a bigger model or a larger context window does NOT fix them, only splitting the work does. Wrong answers almost always offer more capability where the real fix is structure (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/task-decomposition-strategies).
- Narrow-decomposition risk: splitting a broad topic too narrowly yields gaps even when every subagent succeeds (e.g., "AI impact on creative industries" split only into visual-arts subtasks silently omits music, writing, film). Root cause is the decomposition scope, not tool or synthesis failure (https://www.anthropiccertifications.com/learn/agentic-architecture/narrow-decomposition-risk).
- Guard against it: require domain enumeration first (list all relevant sub-domains, then make subtasks against that list), add a coverage-validation step after decomposition, and partition assignments by distinct subtopic/source type per agent to cut overlap (https://www.anthropiccertifications.com/learn/agentic-architecture/narrow-decomposition-risk).
- Diagnostic rule: if all subagents succeed but the aggregate is incomplete, suspect the coordinator's decomposition, not execution (https://www.anthropiccertifications.com/learn/agentic-architecture/narrow-decomposition-risk).
- Route by complexity: send each task to the cheapest model meeting the quality bar; use a lightweight model (Haiku) as a classifier that gates escalation to larger models, and monitor per-tier quality to validate routing (https://www.anthropiccertifications.com/learn/agentic-architecture/task-decomposition-routing).

## Multi-step orchestration of multi-concern requests

- A multi-concern request ("charged twice, discount didn't apply, want to cancel") handled sequentially is slow, refetches the same data per concern, and loses context between concerns (https://www.anthropiccertifications.com/learn/agentic-architecture/multi-step-orchestration).
- Fix: decompose into distinct concerns → fetch shared data (e.g., customer record) ONCE and reuse → investigate concerns in parallel → synthesize one unified resolution (https://www.anthropiccertifications.com/learn/agentic-architecture/multi-step-orchestration).
- If single-concern accuracy is high (~94%) but multi-concern degrades (~58%), add few-shot examples showing correct reasoning and tool sequences for multi-concern messages (https://www.anthropiccertifications.com/learn/agentic-architecture/multi-step-orchestration).

## Session state, resumption, forking

- `--resume <session-name>` continues a specific named prior conversation, preserving full history including previous tool results (https://www.anthropiccertifications.com/learn/agentic-architecture/session-state).
- `fork_session` branches from a shared baseline so you can explore divergent approaches independently (e.g., compare two refactor strategies) without losing the original analysis (https://www.anthropiccertifications.com/learn/agentic-architecture/session-state).
- Decision rule: prior context mostly valid → resume; files changed since last session → resume but tell the agent exactly which files changed for targeted re-analysis; prior tool results stale (underlying data changed) → start fresh with an injected structured summary (resuming on stale results makes the agent reason from outdated info); exploring a different approach → fork (https://www.anthropiccertifications.com/learn/agentic-architecture/session-state).
- The single test behind that rule is "is the old context still TRUE?" CLI surface: `claude --continue` (most recent session in cwd), `claude --resume` (picker), `claude --resume <name>`, and `claude --from-pr <n>` (reopen the session behind a PR); fork via `/branch` in-session or `--fork-session` combined with `--continue`/`--resume` (forks group under their root in the picker, a visible family tree) (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/session-state-resumption-forking).
- Stale-context trap: **fork does NOT cure staleness** — a fork copies the baseline as-is, so both branches inherit the same stale reads. And merely resuming and asking the agent to re-read the changed files is insufficient: the stale results still sit in context beside the fresh reads and the model must reconcile two contradictory versions. The only reliable cure is a fresh session + a structured summary that explicitly NAMES the changed files (https://www.anthropiccertifications.com/courses/claude-certified-architect-foundations/session-state-resumption-forking).
